Task Management Manual

A comprehensive guide to the Task Management module in the FSLib Showcase sample.

Overview

The Task Management module demonstrates FSLib's full CRUD lifecycle: listing tasks in a data grid, creating new tasks via a form, editing existing ones, and deleting them. All data is stored in an in-memory repository — no database required.

Good to know: Since this sample uses in-memory storage, all data resets when the server restarts. Think of it as a sandbox where you can experiment freely.

Task List

The main view shows tasks in a Tabulator data grid with these features:

  • Server-side pagination — only one page of data is loaded at a time.
  • Column sorting — click headers to sort ascending or descending.
  • Header filters — type in the filter inputs below Title and Assignee.
  • Color-coded badges — Priority and Status display as colored labels.
  • Inline delete — trash icon in the last column for quick removal.

Task Form

The task form uses FSLib's formRow / formColumn layout helpers for a clean Bootstrap grid arrangement.

  • Title (required) — the task name.
  • Assignee — who is responsible.
  • Priority — dropdown with Low, Medium, High, Critical.
  • Status — dropdown with Open, InProgress, Done, Cancelled.
  • Estimated Hours — numeric spinner (0–1000).
  • Description — free-text area for notes.

Validation runs client-side before submission. The Title field must be filled in.

Data Model

Each task is a Task data class implementing BaseDoc<String>:

  • _id — unique string identifier (auto-generated on create).
  • title, description, assignee — text fields.
  • priority, status — string-typed enum values.
  • estimatedHours — integer.

Architecture

This sample follows FSLib's standard fullstack architecture:

  • commonMain — shared model (Task), filter (ApiFilter), and RPC service interface (ITaskService).
  • jvmMain — Ktor server with InMemoryRepository backend and TaskService implementation.
  • jsMain — KVision frontend with ViewListTask (grid), ViewItemTask (form), and App (routing).

RPC communication between client and server is handled by Kilua RPC with type-safe service interfaces shared across platforms.

FAQ

Q: Why do my changes disappear after restarting?
A: The sample uses InMemoryRepository, which stores data in a ConcurrentHashMap. Data lives only in server memory and is re-seeded on startup.

Q: Can I change the number of rows per page?
A: The default page size is configured in ViewListTask's Tabulator setup. Look for the fsTabulator call to customize pagination options.

Q: How do I add a new field?
A: Add the property to the Task data class in commonMain, add a column in ViewListTask, and a form field in ViewItemTask. That's it!
